# Контракт поведения Аиши

## 1) Режим chat-only
- Стратегия: `LLM-first + tool-calling + grounding`.
- Модель определяет действие в разговоре: ответить, показать локации, услуги, специалистов, контакты/часы, начать запись.
- Любые факты по аккаунту берутся только из контекста БД (локации, услуги, специалисты, профиль).
- Если в ответе модели есть неза-grounded факты о бизнесе, ответ заменяется на grounded-вариант (реальные данные + кнопки).
- Regex в chat-only используется только как fallback, а не как основной маршрут.

## 2) Транзакционный режим
- Стратегия: детерминированный state machine (`booking-flow` / client actions).
- Критичные операции всегда строгие:
- согласие на ПДн
- подтверждение записи
- отмена записи
- перенос записи
- В этих операциях проверка и валидация имеют приоритет над свободной генерацией.

## 3) Политика при отсутствии данных (no data)
- Если данных в аккаунте нет, ассистент отвечает прямо: «Сейчас не вижу этой информации».
- Запрещено выдумывать адреса, цены, график, специалистов, услуги и слоты.
- Дальше должен быть полезный следующий шаг: кнопки с валидными действиями (локации/услуги/специалисты/запись/мои записи).

## 4) Политика при низкой уверенности (low confidence)
- Если уверенность модели низкая или вызов модели/инструмента не удался:
- включается безопасный fallback-маршрут
- ответ короткий и нейтральный
- показываются action-кнопки для продолжения
- Для критичных транзакций (`consent/confirm/cancel/reschedule`) всегда используется детерминированный путь.

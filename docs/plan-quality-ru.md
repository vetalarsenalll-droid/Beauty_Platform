# План проверки проекта (RU)

Ниже — практичный план ревизии качества проекта. Разделы идут от критичных к второстепенным.

## 1) Сборка и типы
- Убедиться, что `npm run build` проходит без ошибок в каждом app/пакете.
- Проверить TypeScript ошибки/строгие места (`any`, `unknown`, `ts-ignore`).
- Зафиксировать критичные типовые баги (ошибки в runtime-ветках, несовпадение типов API).

## 2) Линтер и формат
- Разобрать `eslint` предупреждения:
  - `react-hooks/exhaustive-deps` (важно, может менять поведение).
  - `no-unused-vars` (косметика, но убрать после критичных).
  - `next/no-img-element` (оптимизация производительности).

## 3) Критические пользовательские сценарии
- Онлайн-запись: от выбора услуги до оплаты/завершения.
- AI-ассистент: сценарии по ключевым веткам (запись, смена времени/специалиста, контакты, согласия).
- Авторизация/личный кабинет: вход, профиль, записи.
- Админ/CRM: ключевые формы и статистика.

## 4) Данные и интеграции
- Проверить API контракты: обязательные поля, null-ы, формат дат/телефонов.
- Валидация на сервере и клиенте должна совпадать.
- Поведение при пустых данных/ошибках API.

## 5) Производительность и UX
- Размеры бандлов, LCP, загрузка страниц.
- Реакция на плохой интернет/медленные ответы.
- Поведение лоадеров и пустых состояний.

## 6) Безопасность и права
- Проверка прав на CRM-страницах.
- Валидация параметров API.
- Утечки данных (особенно в публичных маршрутах).

## 7) Регрессия после изменений
- Список ключевых регрессий, которые нужно проверять после каждого релиза.
- Чек-лист ручного тестирования.

## 8) Документация
- Обновить READMEs, технические заметки.
- Описать важные архитектурные решения.
